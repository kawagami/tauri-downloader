1. 判斷：先量，別猜

三個問題，順序不能反。

Q1 瓶頸是哪一種？

類型特徵並行效果執行緒數
延遲綁定
latency-bound		 每筆等外部回應，CPU 閒著 大勝，常 5–10x 遠超核心數（8–64）
頻寬綁定
throughput-bound		 傳輸量已打滿管線 幾乎 0，可能更慢 無意義
CPU 綁定 核心跑滿 線性到核心數 = 核心數
鎖／序列點綁定 大家排隊等同一個資源 0，先拆鎖 —

本案怎麼判的：70 檔讀取 1.6s，總資料量只有 1.1 MB → 換算 0.7 MB/s。任何硬體都遠不止這速度，不是頻寬。除以檔數 = 每檔 23ms；而平均 16 KB 的檔案不可能吃掉 23ms CPU → 是延遲（WSL 9P 協定每次 round trip 的固定成本）。

關鍵算式

總時間 ÷ 筆數 = 每筆延遲　總資料 ÷ 總時間 = 實際頻寬

兩個數字一比，瓶頸類型就出來了。憑感覺選並行度是賭博。

Q2 工作彼此獨立嗎？

任一條不過，就先改設計再談並行：

  • 碰不同資源？70 個檔案各寫各的 ✔。若是「70 筆都 append 到同一個 log 檔」✘
  • 共享可變狀態？只有計數器 → atomic 可解 ✔。若要共讀共寫一個 HashMap → 要鎖，鎖一多就退化成序列
  • 順序是語意的一部分嗎？本案只要求輸出順序穩定（顯示用），不要求執行順序 → 可保序 ✔。若「後面的計算依賴前面的結果」✘
  • 副作用可重入？覆寫檔案冪等 ✔
  • 外部有配額嗎？WSL 沒限；換成 API 就要看 rate limit、DB 連線池上限
本專案反例

AppState.download_cancelled 是單一全域 AtomicBool（CLAUDE.md 已列為已知問題）。多個下載並行時，一個取消會誤殺全部 —— 這就是「共享可變狀態」擋住並行的典型。要並行得先改成 per-task 旗標。

Q3 單位工作夠大嗎？

執行緒建立約 50–100µs。單筆工作 23ms → overhead 佔 0.4%，划算。若單筆只要 5µs（純記憶體運算），並行反而虧。

2. 量測手法

本案實際跑的四步，可直接套用到別的地方：

// 1. 分段計時,先找出時間花在哪一段
let t = std::time::Instant::now();
let files = collect_files(&roots)?;
println!("collect: {} files in {:?}", files.len(), t.elapsed());

量完發現：collect_files 1766ms、讀檔 1596ms —— 兩段都要打，只打一段等於白做（Amdahl 定律：只優化佔 50% 的部分，整體最多快 2x）。

步驟結果
2. 序列基準線1596ms
3. 並行版（8 條）280ms
4. 掃並行度找拐點4 → 546ms　8 → 345ms　16 → 363ms
拐點的意義

16 條沒有更快，代表已經打到別的瓶頸（連線數、伺服器端處理能力）。再加只是浪費資源、增加對方負擔。找到拐點就停。

量測程式寫成 #[ignore] 測試，量完就刪 —— 它依賴特定環境（真實 WSL 路徑），留在 repo 會變成別人的 CI 紅燈。

本案最終成果

目錄走訪 collect_files 1766ms → 135ms
讀取 70 檔 1596ms → 280ms
預覽總計 3.2s → 0.4s

3. 改寫：三種骨架

型態 A：固定清單的 map

本案的讀檔／寫檔。需求：N 筆已知工作、要保序、各筆耗時不均。

fn par_map<T, R, F>(items: &[T], f: F) -> Vec<R>
where T: Sync, R: Send, F: Fn(usize, &T) -> R + Sync,
{
    let slots: Vec<Mutex<Option<R>>> = items.iter().map(|_| Mutex::new(None)).collect();
    let next = AtomicUsize::new(0);            // 工作分派游標
    let threads = IO_THREADS.min(items.len()); // 別開比工作還多的執行緒

    std::thread::scope(|s| {                   // scope = 可借用外部資料,不用 'static
        for _ in 0..threads {
            s.spawn(|| loop {
                let i = next.fetch_add(1, Ordering::Relaxed);   // 搶號碼牌
                if i >= items.len() { break; }
                *slots[i].lock().unwrap() = Some(f(i, &items[i])); // 寫回自己的格子
            });
        }
    });                                        // scope 結束 = 全部 join 完

    slots.into_iter().filter_map(|m| m.into_inner().unwrap()).collect()
}

三個設計點：

  1. atomic 游標而非預先切塊。切塊（thread 0 拿 0–8、thread 1 拿 9–17…）遇到耗時不均會有人早早閒著。搶號碼牌是最簡單的 work stealing。
  2. slot 寫回保序。輸出順序 = 輸入順序，結果跟序列版完全一樣 → 呼叫端不用改、測試可逐項比對。若丟進共用 Vec push，順序會隨機，UI 每次刷新排序都跳。
  3. thread::scope（Rust 1.63+）。舊寫法 thread::spawn 要求 'static，得把資料 Arc 包起來；scope 保證執行緒在借用結束前 join 完，可以直接借 &items。

型態 B：工作量未知的樹

本案的目錄走訪。需求：一開始只有 2 個根目錄，走著走著長出 371 個，無法預先分派。

let queue = Mutex::new(start);        // 待處理目錄
let active = AtomicUsize::new(0);     // 正在處理中的數量 ← 關鍵
let out = Mutex::new(Vec::new());

std::thread::scope(|s| {
    for _ in 0..threads {
        s.spawn(|| loop {
            let dir = queue.lock().unwrap().pop();
            let Some(dir) = dir else {
                // 佇列空 ≠ 做完!別人手上那筆可能還會吐出新子目錄
                if active.load(Ordering::SeqCst) == 0 { break; }
                std::thread::yield_now();
                continue;
            };
            active.fetch_add(1, Ordering::SeqCst);
            // ... read_dir,子目錄 push 回 queue,檔案 push 到 out
            active.fetch_sub(1, Ordering::SeqCst);
        });
    }
});
最容易寫錯的地方

終止條件。只看「佇列空了就結束」會提早收工：A 執行緒正在讀某目錄、還沒把子目錄放回佇列，此時 B 看到空佇列就跑掉 → 漏檔，而且是隨機漏，測試不一定抓得到。必須 queue.is_empty() && active == 0 兩個條件同時成立。

active.fetch_add 必須在 pop 之後、放開判斷之前，順序錯了一樣有窗口。

這裡用 SeqCst 而非 Relaxed：終止判斷牽涉兩個變數（queue 狀態、active 值）的相對順序，需要強保證；純計數（進度回報）才可以用 Relaxed。

順手撿到的免費加速

entry.file_type() 取代 path.is_dir()。後者每個項目多一次 stat round trip；前者用 read_dir 已經帶回來的資訊。這一項的貢獻不輸並行本身。

教訓：並行前先看看有沒有「根本不必做的 IO」，那是零風險的加速。

型態 C：async IO

本專案 http_dl 用的。程式已在 async runtime、工作是網路 IO 時，不要開執行緒。

use futures_util::stream::{self, StreamExt};

let results = stream::iter(urls)
    .map(|u| async move { fetch(u).await })
    .buffer_unordered(8)   // 同時最多 8 個 in-flight
    .collect::<Vec<_>>().await;

差別：執行緒是 OS 資源（每條約 1 MB stack），async task 是使用者態的、開幾千個沒關係。但檔案 IO 在 Windows/Linux 上 tokio 其實也是丟到 blocking pool，所以本案用 thread::scope 沒有比較差，還少一層抽象。

工作性質選擇
網路 / socketasync（buffer_unordered、join_all）
本地或網路檔案系統執行緒池（thread::scope）
純 CPUrayon 或執行緒數 = 核心數

4. 接回 async 世界

Tauri command 是 async。CPU／blocking 工作直接寫在裡面會卡住 runtime 的 worker：

let worker_app = app.clone();
let result = tauri::async_runtime::spawn_blocking(move || {
    plan_files(&worker_app, &roots, &hall, &codes, &envs)
}).await.map_err(|e| e.to_string())?;

emit_progress(&app, "done", 0, 0, "");   // clone 一份留給外面收尾
result

spawn_blocking 的 closure 要 'static → 所以先 clone 再 move。留一份在外面，是為了不論 Ok/Err 都保證發出 done 事件，前端的進度條才一定收得掉。

5. 並行後要跟著改的東西

容易漏掉的三處：

① 進度語意

序列時「第 12 個檔：xxx.php」是準的；並行下同時有 8 個在跑，「當前檔名」只是抽樣。所以要改成 atomic 累加的完成數才可信，檔名只當視覺回饋。

這點值得寫進註解 —— 不然下一個人會以為是 bug。

② 錯誤語意

序列版可以「一錯就停」；並行版當你發現錯誤時，其他 7 條已經做完了。本案的選擇：

  • per-item error（讀檔／寫檔）—— 每筆自己帶錯誤訊息，不中斷別人
  • 記第一個錯誤最後回報（目錄走訪）

要真正的 fail-fast，就得額外做取消旗標。

③ 測試

並行 bug 是機率性的，一般測試不保證抓到。本案加了 parallel_walk_finds_nested_files：建巢狀暫存目錄，斷言撈齊且排序後與預期完全一致。

這也是型態 A 選擇保序輸出的附加價值 —— 可以直接跟序列版逐項比對。

6. 什麼時候不該並行

  • 順序即語意 —— 例如 reorder_tasks 寫 sort_order，順序錯了資料就錯
  • 對方不歡迎 —— 爬站台並行等於變相攻擊，本專案網站下載刻意保持序列（還有限速 sleep）
  • 共享的單一旗標／連線 —— 前述 download_cancelled，並行前得先重構
  • 省下來的時間人感覺不到 —— 200ms → 40ms 沒人在乎，複雜度卻永久留著
本案值得做的具體理由

環境勾選每改一次就重跑一次預覽，3.2 秒的等待會反覆發生。若只是一次性操作，建議別改。
